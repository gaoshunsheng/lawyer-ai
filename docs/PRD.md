## 一、产品概述

### 1.1 产品定位

律智通是一款面向执业律师的AI法律工作台，专注劳动法领域，覆盖智能咨询、案件管理、文书撰写、合同审查、赔偿计算、庭审模拟全流程，让律师从重复劳动中解放出来。

### 1.2 核心价值主张

| 价值点 | 描述 | 量化目标 |
|--------|------|----------|
| 效率提升 | 文书撰写时间缩短60%以上 | 从平均4小时降至1.5小时 |
| 质量保障 | 法条引用准确，案例检索全面 | 法条引用准确率 > 99% |
| 决策支持 | 基于数据的案件分析预测 | 胜诉预测准确率 > 75% |
| 洞察分析 | 多维度数据报表驱动决策 | 报表覆盖办案全维度 |

### 1.3 目标用户画像

#### 主要用户：执业律师（资深）

- **特征**：5年以上执业经验，独立承办案件
- **需求**：高效的法律研究、精准的案例分析、快速的文书生成
- **痛点**：重复性工作多、案例检索耗时、知识难以沉淀

#### 次要用户：律师助理

- **特征**：1-3年经验，辅助主办律师
- **需求**：文书模板、案件进度跟踪、证据整理
- **痛点**：经验不足、需要频繁请教

#### 辅助用户：律所管理人员

- **特征**：负责律所运营管理
- **需求**：案件统计、团队协作监控、数据分析
- **痛点**：信息分散、难以统筹

#### 向量存储设计（pgvector）

```sql
-- 法规向量表
CREATE TABLE law_embeddings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    law_id UUID NOT NULL REFERENCES laws(id),
    article_number VARCHAR(50),          -- 条款号
    chunk_text TEXT NOT NULL,            -- 分块文本
    chunk_type VARCHAR(20) DEFAULT 'article', -- article/section/full
    embedding vector(1536) NOT NULL,     -- pgvector 1536维
    metadata JSONB DEFAULT '{}',         -- 法规名/效力级别/时效状态
    created_at TIMESTAMPTZ DEFAULT now()
);

-- HNSW索引（高性能近似最近邻）
CREATE INDEX idx_law_embeddings_hnsw
    ON law_embeddings USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- 案例向量表
CREATE TABLE case_embeddings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id UUID NOT NULL REFERENCES precedent_cases(id),
    chunk_text TEXT NOT NULL,
    chunk_type VARCHAR(20) DEFAULT 'ruling',  -- ruling/fact/reasoning
    embedding vector(1536) NOT NULL,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_case_embeddings_hnsw
    ON case_embeddings USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

#### 文本分块策略

| 文档类型 | 分块策略 | 块大小 | 重叠 | 说明 |
|----------|----------|--------|------|------|
| 法律法规 | 按条款分块 | 自然边界 | 无 | 每条法规作为独立chunk，保留条款号 |
| 司法解释 | 按条款分块 | 自然边界 | 无 | 同上 |
| 裁判案例 | 语义分块 | 500字符 | 50字符 | RecursiveCharacterTextSplitter |
| 内部文档 | 语义分块 | 500字符 | 50字符 | 同上 |

#### LangChain核心组件

| 组件 | 选型 | 说明 |
|------|------|------|
| VectorStore | `langchain_community.vectorstores.PGVector` | 直接对接 Supabase pgvector |
| Embeddings | `langchain_community.embeddings.HuggingFaceEmbeddings` 或 智谱 Embedding API | BGE-Large-ZH（本地）或 Embedding-3（智谱API） |
| TextSplitter | `langchain_text_splitters.RecursiveCharacterTextSplitter` | 语义分块，中英文边界感知 |
| Retriever | 自定义 `MultiVectorRetriever` | 向量检索 + 全文检索混合 |
| LLM | `langchain_openai.ChatOpenAI` | OpenAI兼容接口，统一对接智谱AI |
| Output Parser | `langchain_core.output_parsers.PydanticOutputParser` | 结构化输出解析 |

#### 检索增强策略

**混合检索**：向量相似度（语义匹配）+ 全文检索（关键词精确匹配），通过 Reciprocal Rank Fusion (RRF) 融合排序。

**时效性加权**：法规按效力级别加权（法律 > 司法解释 > 部门规章 > 地方法规），过期法规自动降权。

**上下文窗口控制**：注入LLM的检索结果总长度不超过 4000 tokens，优先保留高相似度+高权威性的片段。

**缓存层**：常见问题（如"N+1赔偿标准"）的检索结果缓存到 Upstash Redis，TTL 24小时，减少重复向量计算。

#### LangGraph StateGraph 设计

**全局状态定义**

```python
from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, END

class AgentState(TypedDict):
    # 输入
    query: str                          # 用户原始输入
    session_id: str                     # 会话ID
    case_id: str | None                 # 关联案件ID
    tenant_id: str                      # 租户ID

    # 路由
    intent: str | None                  # 识别的意图
    agent_route: str | None             # 路由目标Agent

    # RAG上下文
    retrieved_laws: list[dict]          # 检索到的法规
    retrieved_cases: list[dict]         # 检索到的案例
    context_text: str | None            # 拼接后的上下文

    # 对话历史
    messages: list[dict]                # 对话消息列表

    # 输出
    response: str | None                # 最终回复
    structured_output: dict | None      # 结构化输出（法律依据/建议等）
    tokens_used: int                    # Token消耗
```

**核心工作流**

```python
# Router → 专业Agent → 工具调用 → 生成回复
workflow = StateGraph(AgentState)

# 节点定义
workflow.add_node("router", router_node)           # 意图识别
workflow.add_node("rag_retrieve", rag_retrieve_node) # RAG检索
workflow.add_node("consult", consult_agent_node)    # 咨询Agent
workflow.add_node("document", document_agent_node)  # 文书Agent
workflow.add_node("review", review_agent_node)      # 审查Agent
workflow.add_node("trial", trial_agent_node)        # 庭审Agent
workflow.add_node("analysis", analysis_agent_node)  # 分析Agent
workflow.add_node("respond", respond_node)          # 响应生成

# 边定义
workflow.set_entry_point("router")

# 条件路由: Router根据意图分发
workflow.add_conditional_edges("router", route_by_intent, {
    "consult": "rag_retrieve",
    "document": "document",
    "review": "review",
    "trial": "trial",
    "analysis": "rag_retrieve",
})

# RAG检索后进入对应Agent
workflow.add_conditional_edges("rag_retrieve", route_after_rag, {
    "consult": "consult",
    "analysis": "analysis",
})

# 所有Agent完成后统一响应
for agent in ["consult", "document", "review", "trial", "analysis"]:
    workflow.add_edge(agent, "respond")

workflow.add_edge("respond", END)
```

| 策略 | 实现 | 说明 |
|------|------|------|
| 行级安全 | PostgreSQL RLS 按 `tenant_id` | 所有查询自动注入租户过滤 |
| 层级查询 | `path` 前缀匹配 | `WHERE path LIKE '/tenant-id/%'` 查询部门树 |
| API隔离 | FastAPI中间件注入 `tenant_id` | 从JWT提取租户信息，所有Service层强制过滤 |
| 文件隔离 | Supabase Storage 策略 | 按租户分桶，RLS控制访问 |

### 4.7 特性C：Token监控与模型管理

#### 功能描述

全层级Token消耗追踪与预算控制，平台统一管理模型提供方，按场景配置最优模型路由。

#### Token消耗追踪

每次LLM调用记录明细：

```sql
CREATE TABLE token_usages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id),
    department_id UUID REFERENCES departments(id),
    user_id UUID NOT NULL REFERENCES users(id),
    session_id UUID,
    model_provider VARCHAR(50) NOT NULL,          -- zhipu / openai / custom
    model_name VARCHAR(100) NOT NULL,             -- glm-5.1 / glm-5-turbo / glm-4-flash
    agent_type VARCHAR(50),                       -- consult / document / review / trial
    input_tokens INTEGER NOT NULL,
    output_tokens INTEGER NOT NULL,
    total_tokens INTEGER NOT NULL,
    latency_ms INTEGER,
    cost_cny DECIMAL(10,6),                       -- 估算成本（人民币元）
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 按日聚合表（加速仪表盘查询）
CREATE TABLE token_usage_daily (
    date DATE NOT NULL,
    tenant_id UUID NOT NULL,
    department_id UUID,
    user_id UUID,
    model_name VARCHAR(100),
    agent_type VARCHAR(50),
    total_input_tokens BIGINT DEFAULT 0,
    total_output_tokens BIGINT DEFAULT 0,
    total_tokens BIGINT DEFAULT 0,
    total_cost_cny DECIMAL(10,4) DEFAULT 0,
    request_count INTEGER DEFAULT 0,
    PRIMARY KEY (date, tenant_id, department_id, user_id, model_name, agent_type)
);
```

#### 全层级预算控制

| 层级 | 预算字段 | 超额策略 |
|------|----------|----------|
| 租户 | `tenants.token_budget_monthly` | ① 降级到GLM-4-Flash → ② 暂停AI功能 → ③ 通知平台管理员 |
| 部门 | `departments.token_budget_monthly` | ① 通知部门管理员 → ② 限制部门AI调用频率 |
| 用户 | `users.token_budget_monthly` | ① 降级到GLM-4-Flash → ② 限制为免费版额度（5次/日） |

**预算检查流程**

```
LLM请求 → 检查用户预算 → 检查部门预算 → 检查租户预算
              │               │               │
           不足:降级       不足:通知       不足:暂停
              ↓               ↓               ↓
              └───────────────┴───────────────┘
                              ↓
                     预算充足 → 执行LLM调用 → 记录token_usage → 累加日聚合表
```
